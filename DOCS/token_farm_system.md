# Система автономного обновления токенов авторизации на базе Android-устройства

## Концепция

Система представляет собой автономную ферму обновления токенов авторизации, развёрнутую на одном физическом Android-устройстве с root-доступом. Каждый токен привязан к отдельному пользовательскому профилю Android, что обеспечивает полную изоляцию данных между аккаунтами. Для целевого сервиса (Avito) каждый профиль выглядит как независимое реальное устройство.

---

## Аппаратная база

**Устройство:** OnePlus 8T (или аналогичный флагман)

- Процессор: Snapdragon 865 (достаточно для последовательного запуска)
- ОЗУ: 12 GB (в моменте работает только один экземпляр приложения)
- Хранилище: 256 GB (~300–500 MB на профиль, до 100+ профилей)
- Root-доступ: Magisk
- Реальное железо: датчики, GPS, гироскоп, модем, baseband — отсутствие любых признаков эмуляции

---

## Архитектура системы

### 1. Пользовательские профили Android

Android поддерживает множественные пользовательские профили на уровне ОС. Каждый профиль — это полностью изолированная песочница с отдельной файловой системой:

```
/data/user/0/com.avito.android/    ← основной пользователь
/data/user/10/com.avito.android/   ← профиль аккаунта #1
/data/user/11/com.avito.android/   ← профиль аккаунта #2
/data/user/12/com.avito.android/   ← профиль аккаунта #3
...
/data/user/N/com.avito.android/    ← профиль аккаунта #N
```

Данные между профилями **физически не пересекаются**: токены, cookies, shared_preferences, базы данных — всё хранится в отдельных директориях.

**Снятие лимита профилей (по умолчанию 4–5):**

```bash
setprop fw.max_users 100
echo "fw.max_users=100" >> /system/build.prop
```

### 2. Управление жизненным циклом токенов

Система не использует последовательный обход всех профилей. Вместо этого каждый аккаунт имеет **индивидуальное расписание**, привязанное к времени истечения его токена авторизации.

**Алгоритм:**

1. Система знает время истечения токена каждого аккаунта
2. За 1 минуту до истечения запускается приложение от имени соответствующего user profile
3. Сессия поддерживается ~2 минуты — токен обновляется автоматически
4. Frida перехватывает новый токен и сохраняет/отправляет на сервер
5. Приложение закрывается

**Расписание (cron):**

```
16 3  * * * /data/local/tmp/refresh.sh 10    # аккаунт #1 — токен истекает в 03:17
41 8  * * * /data/local/tmp/refresh.sh 11    # аккаунт #2 — токен истекает в 08:42
02 14 * * * /data/local/tmp/refresh.sh 12    # аккаунт #3 — токен истекает в 14:03
...
```

### 3. Скрипт обновления токена

```bash
#!/system/bin/sh
# refresh.sh — принимает user_id как аргумент
UID=$1
PACKAGE="com.avito.android"
ACTIVITY="$PACKAGE/.MainActivity"
FRIDA_SCRIPT="/data/local/tmp/grab_token.js"

# Запуск приложения от имени конкретного профиля
am start --user $UID -n $ACTIVITY
sleep 90  # держим сессию ~1.5 минуты для обновления токена

# Frida перехватывает новый токен
PID=$(ps -A | grep $PACKAGE | grep "u${UID}_" | awk '{print $2}')
if [ -n "$PID" ]; then
    frida -U -p $PID -l $FRIDA_SCRIPT --no-pause
    sleep 30
fi

# Закрытие приложения
am force-stop --user $UID $PACKAGE
```

### 4. Перехват токенов через Frida

Frida подключается к процессу приложения и перехватывает токены авторизации на лету. Скрипт `grab_token.js` хукает сетевые вызовы или хранилище токенов и сохраняет актуальные значения.

## Маскировка и антидетект

### Что уже изолировано на уровне ОС

| Параметр | Изолирован? |
|----------|-------------|
| Android ID | ✅ Уникален для каждого профиля |
| Данные приложения (токены, cookies) | ✅ Отдельная файловая система |
| Google Advertising ID | ✅ При разных Google-аккаунтах |
| Список установленных приложений | ⚠️ Частично (свой набор, но ядро общее) |

### Что одинаково и может потребовать подмены

| Параметр | Подмена |
|----------|---------|
| IMEI | Через Frida — генерация валидного IMEI с корректным TAC для OnePlus 8T и контрольной цифрой по алгоритму Луна |
| Build.MODEL / MANUFACTURER | Оставляем OnePlus 8T (реалистично) |
| MAC-адрес | Через Frida-хук NetworkInterface |
| SIM-оператор / номер | Через Frida-хук TelephonyManager |
| WiFi SSID / BSSID | Через Frida-хук WifiInfo |
| IP-адрес | Ротация через резидентные прокси |

### Генерация уникального IMEI

Первые 8 цифр (TAC) остаются от OnePlus 8T, серийная часть генерируется уникально для каждого профиля, контрольная цифра вычисляется по алгоритму Луна:

```javascript
var TAC = "86776104";  // реальный TAC OnePlus 8T

function luhnCheckDigit(partialImei) {
    var sum = 0;
    for (var i = 0; i < partialImei.length; i++) {
        var digit = parseInt(partialImei[i]);
        if ((partialImei.length - i) % 2 === 0) {
            digit *= 2;
            if (digit > 9) digit -= 9;
        }
        sum += digit;
    }
    return ((10 - (sum % 10)) % 10).toString();
}

function generateIMEI(userId) {
    var serial = "";
    for (var i = 0; i < 6; i++) {
        serial += ((userId * 7919 + i * 104729) % 10).toString();
    }
    var partial = TAC + serial;
    return partial + luhnCheckDigit(partial);
}
```

### Подход к подмене

**Прагматичная стратегия:**

1. Запустить Frida-сниффер, который логирует все системные API-вызовы приложения
2. Определить, какие именно параметры Avito реально считывает
3. Подменять только те параметры, которые фактически запрашиваются

Нет смысла хукать 50 API, если приложение использует только 5.

## Разведка: определение fingerprint-профиля Avito

Перед настройкой подмены необходимо выяснить, какие данные Avito реально считывает с устройства. Для этого используется Frida-скрипт, который хукает все основные системные API и логирует вызовы:

- `TelephonyManager` — IMEI, IMSI, SIM serial, оператор, номер телефона
- `Settings.Secure` — Android ID и другие системные настройки
- `NetworkInterface` — MAC-адрес
- `WifiInfo` — SSID, BSSID точки доступа
- `Build` — модель, производитель, серийный номер
- `AdvertisingIdClient` — Google Advertising ID
- `LocationManager` — GPS-координаты
- `PackageManager` — список установленных приложений
- `AccountManager` — Google-аккаунты на устройстве
- `SensorManager` — список датчиков
- `FileInputStream` — прямое чтение /proc/cpuinfo и /sys/

Запуск со spawn для перехвата с момента старта:

```bash
frida -U -f com.avito.android -l sniff_fingerprint.js --no-pause
```

## Преимущества перед альтернативными решениями

### По сравнению с Redroid-фермой (эмуляция)

| Критерий | Redroid | Android-устройство |
|----------|---------|-------------------|
| Реальные датчики | ❌ Эмуляция | ✅ Настоящие |
| Модем / baseband | ❌ Отсутствует | ✅ Реальный |
| Признаки виртуализации | ⚠️ Обнаружимы | ✅ Отсутствуют |
| Стоимость инфраструктуры | Сервер + трафик | Телефон + электричество |
| Масштабирование | Горизонтальное (больше серверов) | Вертикальное (больше профилей) |
| Сложность обнаружения | Средняя | Минимальная |

### По сравнению с реверсом API напрямую

| Критерий | Прямые запросы | Через устройство |
|----------|---------------|-----------------|
| Обнаружение | Высокое (паттерны запросов) | Минимальное |
| Поддержка при обновлениях | Реверс заново | Приложение обновляется само |
| Генерация токенов | Нужно воспроизвести логику | Приложение делает всё само |

## Ресурсы и ограничения

**Потребление хранилища:**
- ~300–500 MB данных на профиль
- 100 профилей ≈ 30–50 GB из 256 GB доступных

**Потребление ОЗУ:**
- В моменте работает только один экземпляр → ~500 MB–1 GB
- Остальные 11 GB свободны

**Потребление CPU:**
- Приложение активно 2 минуты на аккаунт
- 100 аккаунтов с распределёнными по суткам расписаниями → телефон простаивает 95%+ времени

**Электропитание:**
- Телефон на постоянной зарядке
- Отключено агрессивное энергосбережение OnePlus

## Развёртывание

### Первоначальная настройка

1. Установить Magisk (root)
2. Установить Frida-server на устройство
3. Поднять лимит пользователей: `fw.max_users=100`
4. Создать необходимое количество user profiles
5. Установить Avito на каждый профиль
6. Авторизовать каждый аккаунт вручную (один раз)
7. Настроить cron-расписание для каждого профиля

### Создание профилей

```bash
for i in $(seq 1 50); do
    pm create-user "avito_acc_$i"
done

# Установить приложение на все профили
for uid in $(pm list users | grep -oP 'UserInfo\{\K[0-9]+' | grep -v '^0$'); do
    pm install --user $uid /path/to/avito.apk
done
```

### Автозапуск

Через Magisk-модуль (`/data/adb/service.d/`) или busybox crond для запуска скриптов обновления по расписанию.

## Итог

Система превращает одно Android-устройство в автономный, практически необнаружимый сервер обновления токенов. Реальное железо, изолированные профили ОС и точечная подмена fingerprint делают каждый аккаунт неотличимым от обычного пользователя с настоящим телефоном.
